Arch Linux Installation Tutorial

Until about 6 months ago, I’d never really considered linux to be an operating system I’d use in the future - Windows was familiar to me, I knew how to make programs work in it, and linux just seemed to difficult to get working.
But eventually I chose to bit the proverbial bullet and switched my PC over from Windows 11 to Garuda, and I was immediately hooked. I later switched to Arch on that computer, and having recently fixed an old laptop of mine I’ve installed it on that too. The one thing that did get me though was that I was following two tutorials, one from the official Arch wiki and another from phoenixNAP, with a variety of other websites to set up the rest of my operating system.
So, I decided to inaugurate my blog with a tutorial on how I installed and set up Arch, let’s go!
My desktop on Arch

My Setup

The great thing about Arch is that you get a very minimal setup upon install - a command line is pretty much all you get. On the one hand, it can make it somewhat daunting to setup, but on the other hand, you can customise it pretty much however you want. I’ll be showing you how to install arch and set it up like mine, but you can totally change some things around to suit your preferences! If you’re a first time linux user, I suggest trying out a less intensive distro first - Garuda is quite good for gaming on Linux, but if you’re not so bothered about that, try out Manjaro or Ubuntu. One thing to note however is that Ubuntu is based on Debian, but Garuda and Manjaro are based on Arch, so they’re a little different.

What you’ll need

Arch install media

You can download a .iso of arch linux from the official website and burn it onto a CD or USB. The tools I’ve used for this are Rufus on Windows, GNOME Disk Utility for Linux GNOME users, and Popsicle for all Linux users.

An Internet Connection

While it is possible to configure WiFi networks in the Arch Install Media, I’ve never done it. I opt to use an ethernet cable plugged into the back of my PC or Laptop - the live environment should have all the correct drivers installed so you don’t need to worry much about getting onto the internet.

Backups

Installing Arch on a drive will destroy all data already on that drive. As such, it’s very important to either make a backup of your drive or at least make a copy of anything you want off the drive (think pictures, songs, documents, configurations, etc.).

Installing Arch

N.B. I’ll be doing this in QEMU, so the Machine and View windows you see at the top won’t be there for you.

You should now be ready to install Arch on your computer. Turn your computer off, plug in your install media, turn your computer on, and enter the BIOS to change the boot order. Accessing BIOS varies between motherboard manufacturers, but typically you’ll want to press either Del or one of the F1-F12 keys - check your motherboard manual or their website for more information on how to do this. Once in the BIOS, change the boot order to boot from USB or Diskette drive first (whichever your install media is on). You’ll be greeted by a menu providing different options, press enter with Arch Linux Install Medium selected.
Boot options for the install media. Yours might look different to this if you booted in UEFI mode.

After a while, you’ll be greeted with a shell. This is where we’re going to be inputting a slew of commands, so get ready for some typing. Be careful though, misclicks could be a serious problem.
The arch shell you’ll be greeted with after loading.

Pre-install checks

First comes the simple but important step to make your life way easier - loading up the locale. Because keyboards differ between languages and regions, you might find that trying to type in one character comes out as a completely different character on the screen. I remedy this by running:

loadkeys uk

If you want a different locale for a different keyboard, try running

ls /usr/share/kbd/keymaps/**/*.map.gz

and pick out the locale name you want, then run the loadkeys command with your choice.

Since you need an internet connection, it’s a good idea to check it’s present before you’ve done lots of work. Run

ping somecollagist.github.io

and check for responses. If this fails, check your ethernet cables and internet connection.

I suggest checking to see if your system supports UEFI booting. UEFI is a more secure way of booting operating systems which is included on most modern hardware. Keep in mind that UEFI-enabled systems are still capable of booting operating systems in BIOS mode, which makes this a good choice for security. This article will outline how to install arch on both UEFI drives and BIOS drives. You can check how your computer booted your install medium by running

ls /sys/firmware/efi/efivars

and if there are no errors reported, you’re in UEFI mode.

Finally, check that your system clock is correct. Do this by running

timedatectl set-ntp true && timedatectl status

and if it’s the correct time (it’s in UTC remember), you’re all good to go.

Preparing the drives

Here comes the scary bit. We need to install arch on our drive, which will likely require the formatting and wiping of the entire disk - this isn’t strictly necessary but I would suggest it as dualbooting is beyond the scope of this article. This article will assume you will wipe your drive entirely.

Let’s start by listing the disks present. Run

fdisk -l

to see which storage devices are currently connected to the computer. Identify which drive you want to install Arch on (probably by it’s storage size) - it’ll probably be called something like /dev/sda or /dev/nvme0n1. For the purposes of demonstration, this article will say you want to install Arch on /dev/sda.
fdisk -l

Setting up drives for BIOS

On BIOS, you want to create a DOS partition table for your drive. Run

fdisk /dev/sda

and press o (this creates a DOS partition table), enter, then w to write this to the disk. Now, run

cfdisk /dev/sda

to create the disk partitions. We only need one partition on BIOS, but most users allocate a Linux Swap partition, which is where virtual memory will be stored.

The typical partition layout for a BIOS drive is as follows:

Type Size What we’ll call it
Linux Swap At least 512MiB [ROOTPART]
Linux The rest of your device [SWAP]

Setting up drives for UEFI

On UEFI, you want to create a GPT partition table for your drive. Run

fdisk /dev/sda

and press g (this creates a GPT partition table), enter, then w to write this to the disk. Now, run

cfdisk /dev/sda

to create the disk partitions. We only need two partition on GPT, but most users allocate a Linux Swap partition, which is where virtual memory will be stored.

The typical partition layout for a UEFI drive is as follows:

Type Size What we’ll call it
EFI System At least 300 MiB [EFIPART]
Linux Swap At least 512MiB [ROOTPART]
Linux The rest of your device [SWAP]
An example of creating filesystems on your drive, I’m using a DOS partition table.

Creating partitions in cfdisk

  1. Create a new partition by selecting the new tab at the bottom with the arrow keys.

  2. Press enter and input a size for your drive.

  3. Press enter, select the primary option, and press enter again.

  4. With the partition selected, select the type tab at the bottom with the arrow keys.

  5. Press enter, select which partition type you need, and press enter again.

  6. Repeat for all the other partitions you need, ensuring you select free space for new partitions. If you’re using a BIOS drive, for your root partiton [ROOTPART], make sure you select the bootable tab and press enter.

  7. Write your partitions to the drive, this cannot be undone! Type ‘yes’ to confirm this. Keep note of the device names, such as /dev/sda1 and what partitions they correspond to - whenever I write [ROOTPART], you should write the corresponding device name. You can then quit the cfdisk dialog.

Assigning filesystems

Once you’ve created the partitions for your drive, you need to give them filesystems in order to actually work.

If you created an EFI partition, runYours might look different to this if you booted in UEFI mode.

For your Linux partition, run

mkfs.ext4 [ROOTPART]

to create an ETX4 partition. Keep in mind that if you would rather use a different format, like BTRFS, you should run the relevant command instead. This should be done now.
An example of creating filesystems on your partitions

Mounting drives

Now we need to ‘mount’ our drives to actually use them and install Arch onto them. This is simple enough; run the corresponding commands for whichever partitions you’ve made:

Partition Name Command
Linux mount [ROOTPART] /mnt
EFI System mount --mkdir [EFIPART] /mnt/boot
Linux Swap swapon [SWAP]
Mount the drives

Getting the reflector

Arch uses pacman to install packages on your arch system, and it uses mirrors (websites which point to databases of packages) in order to do this. Since Arch can be installed this way, it is imperative to set up the reflector now so packages can be collected for installation. The reflector on the install media will be copied over into the final system during installation.

To get started, run

pacman -Syy && pacman -S reflector

to sync the pacman databases and install the reflector package.
Install the reflector package

Once this is done, run

cp /etc/pacman.d/mirrorlist /etc/pacman.d/mirrorlist.bak

to create a backup of the current mirrorlist. The install media does come with a list of mirrors, but it’s best to update it. The above command creates a copy of the current mirrorlist so if something goes wrong, you can restore it. Once you’re ready to update the mirror list, run

reflector -c "XX" -f 12 -l 10 -n 12 --save /etc/pacman.d/mirrorlist

“XX” is a country code - for me that’s GB. to get a list of country codes, run

reflector --list-countries

and use the corresponding country code.

Install Arch

It’s time. To install Arch on a the drive, run

pacstrap /mnt base linux-firmware sudo dhcpcd [text-editor] [kernel]

but replace [text-editor] and [kernel] with packages of your choice. For my text editor, I use nano, and the most popular kernel packages are linux (the basic model) or linux-zen (good for gaming). Once you’ve written this command with a text editor and a kernel included, hit enter and let the install media do it’s thing. This takes some time, so be patient.
Run pacstrap to install Arch
Ideal output of pacstrap

Once this is done, type the command

genfstab -U /mnt >> /mnt/etc/fstab

This is very important, as this will define which drives should be mounted at startup, including the drive where your operating system is stored.

Getting into the installed system

You should now be at a point where you can start configuring your system - in order to do this you must first run this command:

arch-chroot /mnt

Immediately after running this, you may notice that your shell has changed. Instead of using ZSH (the default terminal for arch install media), you are now using Bash. This is because Bash is the default shell for an installed Arch system. Any commands you run now will be performed directly on your root partition [ROOTPART].
arch-chroot /mnt means that your shell’s root directory is now on your mounted drive, not the install media.

Setting up the local information

I usually set the clock first before I customise anything. Run

timedatectl list-timezones

to list all available timezones. They’ll be separated into Region/City, so look for your continent and a nearby city. For me, my timezone is Europe/London, so I run the command

timedatectl set-timezone Europe/London

For the next few steps, I will be modifying text in files with my text editor, nano, but if you’ve chosen to use a different editor that’s totally ok. This tutorial will assume you’re using nano.

Now I suggest setting the locale. This will (hopefully) make your keyboard layout permanent, and is equivalent to setting your computer language in other operating systems. You need to edit the file /etc/locale.gen, so I would run

nano /etc/locale.gen

Once I run this, I’ll be presented with a text file will look a little like this:

All those hash characters (#) on the sides are comment characters. They tell the system to ignore any characters after them on a line, so currently there’s no locale set. Scroll down to your locale (it’ll usually be two lowercase characters for your language and two uppercase characters for your region, mine is en_GB), and remove the hash character before it. Make sure you do this to the one with UTF-8. (See the above picture.) If you aren’t sure which locale you should go with, look up which one is right for you. Make a note of your locale, and in nano, press Ctrl+S (to save) and then Ctrl+X (to exit)

Now we should edit the /etc/locale.conf file; I will run

nano /etc/locale.conf

type in the following

LANG=[locale_name]

and then save and exit this file.
How I set up locale.conf

To finish setting up the locale, run

locale-gen

Adding a user

Up to this point, you’ve been configuring the device as the root user. For those people coming from Windows or MacOS, the root user is Linux’s equivalent of an administrator. This should be secure, so set a root password by running

passwd
Setting the root password - you won’t be able to see what you type

It’s worth creating a seperate user so you have a home directory but don’t worry, you can easily be granted administrator priviledges (or root access if you want to sound fancy). To start, run the command

useradd -m -G wheel -s /bin/bash username
Adding a user

You can change username to whatever you like. The -m option gives you a home directory located at /home/username (though you may see it often written as ~), -G wheel adds you to the wheel group (more on that later), and -s /bin/bash sets your default shell to Bash (you can easily change Bash out for another terminal like Fish or ZSH). You should then set a password by running

passwd username

To get root access, you must modify the sudoers file. This can be done by running the command

export EDITOR=nano & visudo

and decommenting the line %wheel ALL=(ALL:ALL) ALL
Setting up sudo access

Installing the bootloader

This step is crucial - the bootloader is a program run when your computer turns on which is responsible for loading the operating system. Most linux users use GRUB to perform this task, but it differs between if your drive boots in BIOS or UEFI mode. This article assumes your computer uses an x86 or x64 architecture - if you use a different processor type, this article is probably not something you need!
